    Herència especialització

    Herència per especialització en bases de dades

    En bases de dades relacionals es poden utilitzar conceptes similars a l'herència en Programació Orientada a Objectes (POO) per modelar situacions on una entitat és una especialització d'una altra. Això s'aconsegueix mitjançant estratègies d'herència en bases de dades, tot i que la implementació no és tan directa com en POO.

    Opcions per modelar herència a bases de dades relacionals

    En bases de dades relacionals, hi ha diverses maneres de gestionar les especialitzacions, i el mètode que triïs dependrà del teu cas d'ús específic. Les estratègies més comunes són:

    1. Taula única (Single Table Inheritance): Emmagatzemar tots els tipus en una única taula i afegir una columna per identificar el tipus de cada registre.
    2. Taules per a cada classe (Class Table Inheritance): Crear una taula per a la classe base i taules separades per a cada subclasse (com estàs suggerint).
    3. Taula de relació (Concrete Table Inheritance): Crear una taula separada per cada classe (base i subclasse), però amb duplicació de dades entre les taules.

    2.1. Utilitzar herència mitjançant una taula pare

    Si Socis i Administradors són especialitzacions de la mateixa entitat (per exemple, una taula pare anomenada Persones), es pot modelar la relació d'aquesta manera:

    Taula Pare: Persones

    -- Creació de la taula Persones (taula pare)
CREATE TABLE Persones (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nom VARCHAR(255),
    dni VARCHAR(9) UNIQUE,
    tipus ENUM('soci', 'administrador')
);

    Taula Socis (relacionada amb Persones)

    SQL

CREATE TABLE Socis (
    id INT PRIMARY KEY,
    llibres_prestats TEXT,
    FOREIGN KEY (id) REFERENCES Persones(id)
);

    Taula Administradors (relacionada amb Persones)

    SQL
    1
2
3

    -- Creació de la taula Administradors (subclasse)
CREATE TABLE Administradors (
    id INT PRIMARY KEY,
    carrec VARCHAR(50),
    FOREIGN KEY (id) REFERENCES Persones(id)
);

    Explicació:

    • Persones: Conté els camps comuns per a totes les persones, com el nom, dni i un tipus per diferenciar els socis i administradors. Aquesta taula seria la taula pare.
    • Socis i Administradors: Tenen una clau forana (id) que fa referència a la taula Persones. Així, Socis i Administradors són subclases de Persones.

    Consultes:

    Per obtenir els socis o administradors, pots utilitzar una consulta amb una unió:

    SQL
    1
2
3
4
    -- Obtenir tots els socis
SELECT * FROM Persones p
JOIN Socis s ON p.id = s.id
WHERE p.tipus = 'soci';

    Per obtenir tots els administradors:

    SQL
    1
2
3
4
    -- Obtenir tots els administradors
SELECT * FROM Persones p
JOIN Administradors a ON p.id = a.id
WHERE p.tipus = 'administrador';

    2.2. Tipus de relacions d'herència

    1. Taula única (Single Table Inheritance):

    2. Emmagatzemes tots els registres en una sola taula amb una columna que indica el tipus de registre (soci, administrador, etc.).

    3. Avantatge: Simplicitat i eficiència en la consulta.
    4. Desavantatge: Els registres de cada tipus poden estar molt diferents, el que crea molta redundància.

    Exemple de taula única:

    SQL
    1
2
3

    CREATE TABLE Persones (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nom VARCHAR(255),
    dni VARCHAR(9) UNIQUE,
    tipus ENUM('soci', 'administrador'),
    llibres_prestats TEXT, -- només rellevant per a socis
    carrec VARCHAR(50) -- només rellevant per a administradors
);

    Aquesta taula té camps específics per a socis i administradors, però pot contenir registres de tots els tipus, el que pot generar molta redundància.

    1. Taules per a cada classe (Class Table Inheritance):

    2. Crees una taula per a la classe base (en aquest cas Persones) i taules separades per a cada subclasse (Socis i Administradors).

    3. Avantatge: Manté la normalització i evita la redundància.
    4. Desavantatge: Les consultes poden ser més complexes, ja que hauries de fer unió entre taules per obtenir tota la informació.

    Conclusió

    En aquest cas, com que Socis i Administradors tenen atributs diferents, però comparteixen informació comuna (nom i DNI), l'herència mitjançant una taula pare és una bona opció. Així, pots gestionar totes les persones en una taula Persones i crear taules especialitzades per a socis i administradors.

    D'aquesta manera, aconsegueixes evitar la duplicació de dades i mantens una estructura més flexible i escalable.
